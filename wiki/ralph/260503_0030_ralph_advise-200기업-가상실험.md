# advise_vote 200 기업 가상 실험

`advise_vote_before_meeting` tool로 사용자가 던질 수 있는 모든 질문 시나리오를 200 기업에 자동 실행하고 결과를 모은다.

## 가정 (가상환경 명시)
- **컨텍스트 모름**: LLM은 이 프로젝트의 사전 지식 / 대화 히스토리 / 메모리 X
- **웹검색 X**: Naver/Google/뉴스 등 외부 검색 X
- **MCP only**: 오직 OPM MCP server의 17 tool만 사용
- → 순수 MCP 도구만으로 사용자의 의결권 행사 보조 능력을 측정

이 실험은 "MCP가 single source of truth로 충분한가"를 검증하는 stress test.

---

## 유니버스 (200 회사)

### 정기주총 KOSPI 100 + KOSDAQ 50 (=150)
- KOSPI 100: `test/test_kospi200.py`의 첫 100 ticker (시가총액 desc)
- KOSDAQ 50: Naver Finance market cap top 50 (sosok=1)
- 사업연도: 2025 (2025년 정기주총 결과 기준)

### 임시주총 50
- DART list.json `pblntf_ty=E`, "주주총회소집공고", 4월 1일~ 검색
- 회사 dedupe → 첫 50 회사 (검색 정렬 = 공시일 desc)
- 정기는 보통 3월말까지 끝났으니 4월 이후는 사실상 임시주총

## 매 iteration 작업

1. **현황 확인**: `git status` + 진행률 (몇 회사 완료 / 결과 csv 행 수)
2. **batch 호출**: advise_vote_before_meeting을 회사 N개씩 병렬 (4-6 worker)
3. **결과 csv 작성**: 회사별 status / agenda count / 안건별 결정 / 후보 평가 / Marco red_flag
4. **commit** (incremental — 1 batch 끝날 때마다 csv push)
5. 다음 iteration 1줄 plan

## 체크할 시나리오 (advise_vote 가능 모든 질문)

각 회사마다 LLM이 사용자에게 답할 수 있어야 하는 질문들:

### A. 회사 펀더멘털 (financial_summary 필드 활용)
- 매출 / 영업이익 / 순이익 (지배주주 귀속)
- ROE / ROA / 부채비율
- 자본잠식 상태 (normal / partial / 50plus / full)
- 감사의견 (적정 / 한정 / 부적정 / 거절)

### B. 안건별 의결권 결정 (agenda_decisions)
- 안건 카테고리 (financial_statements / director_election / audit_committee / 보수 / 배당 / 정관 / 자사주 등)
- 결정 (FOR / AGAINST / REVIEW)
- 결정 사유 (정책 근거 + 사실 근거)
- vote_style별 차이 (open_proxy vs 운용사별)

### C. 후보 평가 3축 (candidates_evaluations)
- 독립성 (최대주주 관계 / 거래 / 5년 룰)
- 결격사유 (나이 / eligibility)
- Marco 시나리오 (활성 시 — 과거 회사 회계 risk overlap)
- 후보별 추천 사유 (raw text)

### D. 지배구조 + 거버넌스 (ownership / governance)
- 최대주주 + 특수관계인 지분
- 5% 블록 / 자사주
- KCGS 15 지표 준수율

### E. 위임장 분쟁 (proxy_contest 통합 X — recap 영역)
- advise_*에는 없음 — 사전에 분쟁 신호 X
- 단, 후보 평가에서 "주주측 추천" 후보 detect 가능

## 막힘 발생 시
- DART rate limit (1,000/min × 키 2개) → batch worker 4 → 2로 감소
- 회사명 alias 누락 → resolve_company_query fallback 실패는 csv에 status=error 명시
- shareholder_meeting v2 검색 누락 → director_evaluation fallback 활용 (이미 구현)
- 200 회사 batch 60분 timeout → 결과 csv incremental write로 진행률 보존
- → promise X, STATUS REPORT로 결과 + 막힘 영역 보고
